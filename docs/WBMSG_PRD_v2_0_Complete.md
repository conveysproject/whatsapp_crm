# 2. Problem Statement
## 2.1  The Market Failure
Every WhatsApp CRM tool on the market today was built for one of two customers: enterprise teams who need scale, or developers who need APIs. Neither serves the real SMB owner — a shop owner in Pune, a coaching institute in Hyderabad, a boutique brand in Delhi — who just wants to talk to customers and grow.
## 2.2  Pain Points (Validated by Market Research)
### Pain 1: Nobody explains billing clearly
SMBs routinely receive surprise invoices. Message costs, API charges, template submission fees, and monthly platform fees are bundled in ways that make it impossible to predict monthly spend. This destroys trust and causes churn.
### Pain 2: Onboarding is a nightmare
Setting up Meta Business Manager, applying for WhatsApp API access, verifying business identity, submitting message templates, integrating with a third-party tool — the average SMB takes 3-5 days to send their first message. Many give up.
### Pain 3: CRM is an afterthought
Every competitor stores contacts as a flat list with a name and phone number. There are no custom fields, no deal stages, no lifecycle tracking, no notes, no activity history. Businesses still use spreadsheets alongside their WhatsApp tool — defeating the purpose.
### Pain 4: Support is non-existent
When something breaks — a template gets rejected, a number gets flagged, a campaign fails to send — support tickets take 48-72 hours to get a response. For a small business, a 3-day outage is catastrophic.
### Pain 5: Automation is too hard to build
Most tools offer a basic keyword-reply chatbot. Building a real flow — lead capture → qualification → follow-up → sale → post-sale support — requires developer knowledge none of the target SMBs have.
### Pain 6: No intelligence or proactive insights
No existing tool warns you before a customer churns, suggests the best time to follow up, or tells you which contact is most likely to buy again. Businesses operate reactively — they never see patterns in their own data.

## 7.2  Multi-Tenancy Model
WBMSG uses a shared database with strict Row Level Security (RLS) enforced at the PostgreSQL level. Every query is automatically scoped to the authenticated organisation&apos;s ID. This provides cost efficiency, database-enforced isolation, instant tenant provisioning, and architectural impossibility of cross-tenant data access.
## 7.3  Voice AI Architecture
Voice note received via WhatsApp webhook → stored in S3 → transcription job queued in BullMQ
Whisper API called asynchronously (P99 latency target: &lt;5 seconds for notes up to 60 seconds)
Transcript stored in database, linked to the message record
Classification + sentiment analysis run immediately post-transcription using the same Claude API pipeline as text messages
Field extraction runs as a separate low-priority job (within 30 seconds)
Outbound TTS: ElevenLabs API called on-demand when agent triggers voice reply; audio cached for repeated use of the same text
## 7.4  Predictive Analytics Architecture
Separate Python microservice (FastAPI) with its own database connection pool — does not share resources with the real-time API
Model training: nightly batch job trains per-account models for churn prediction and LTV using the previous 90 days of data
Prediction refresh: churn scores and LTV predictions updated daily and written to PostgreSQL; served via REST to the main API
Minimum data requirement: predictions activate after 90 days of account history and a minimum of 100 conversation events
Fallback: if model confidence is below threshold, predictions are hidden and replaced with: &apos;Not enough data yet — predictions will appear after 90 days&apos;
## 7.5  Scaling Strategy
Broadcast campaigns: queued via BullMQ with configurable concurrency (1,000 msg/hour per account default)
Incoming webhooks: stateless webhook processors behind a load balancer
Real-time inbox: Socket.io with Redis adapter for multi-server pub/sub
Database reads: read replicas for analytics and ML queries; primary for writes
AI calls: async with 3-second timeout; graceful fallback if API unavailable
Voice transcription: separate BullMQ queue with dedicated worker pool; does not compete with message delivery
## 7.6  Security
All data encrypted at rest (AES-256) and in transit (TLS 1.3)
API keys hashed with bcrypt, never stored in plaintext
Webhook payloads verified with HMAC-SHA256 signature
SOC 2 Type II roadmap (target: Month 18)
VAPT (Vulnerability Assessment) every 6 months
India DPDP Act compliance built-in from day 1
Voice data: transcriptions stored encrypted; raw audio deleted from S3 after 30 days (configurable)
ML models: trained on anonymised, aggregated patterns only — no raw conversation text used in model training
