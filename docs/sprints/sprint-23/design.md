# Sprint 23 — Trained ML Models

## Sprint Goal
Replace the Sprint 18 heuristic churn and LTV predictors with trained scikit-learn models — making Trust Score and predictions more accurate now that 6 months of historical outcome data is available from beta accounts.

## What We're Building

- **Churn training script** — `services/ml/training/train_churn.py`: reads a historical CSV (`lifecycle_stage`, `engagement_ratio`, `days_since_last_message`, `has_open_deal`, `deal_value`, `churned`), fits a `GradientBoostingClassifier`, prints a classification report, and saves the model to `services/ml/artifacts/churn_model.joblib`.
- **LTV training script** — `services/ml/training/train_ltv.py`: reads a historical CSV (`deal_value`, `engagement_ratio`, `days_since_last_message`, `actual_ltv`), fits a `GradientBoostingRegressor`, prints MAE, and saves to `services/ml/artifacts/ltv_model.joblib`.
- **Updated predictions.py** — `services/ml/models/predictions.py`: `ChurnPredictor` and `LTVPredictor` classes load their respective `joblib` artifact at startup (`__init__`). If the artifact file does not exist (first run, no training data yet), falls back to the Sprint 18 heuristic. Same API contract — no route changes needed.
- **pytest test suite** — `services/ml/tests/`: `conftest.py` with `ContactFeatures` fixtures (active_customer, churned_lead); `test_trust_score.py` (score range, breakdown keys, integer type); `test_predictions.py` (churn labels, LTV positivity). All tests pass against both heuristic (no artifact) and trained (artifact present) paths.

## Key Technical Decisions

- **`GradientBoostingClassifier` over Logistic Regression or Random Forest** — GBM handles imbalanced churn data (few positive examples) better than Logistic Regression and is more interpretable than Random Forest (feature importances). At this dataset size (<10K samples), GBM fits in seconds.
- **Heuristic fallback if no artifact** — The model artifacts are not committed to git (too large; also, they should be versioned separately). On a fresh deploy without artifacts, the service runs the Sprint 18 heuristic. This prevents startup failure and makes the training step optional in CI.
- **Training is a one-time manual step, not CI** — Training requires labelled historical data that doesn't exist in the repo. Engineers run the training script locally with exported data, commit the artifact to an S3 bucket, and configure the ECS task to download it at startup. CI does not train — it only runs `pytest`.
- **Feature vector is the same for heuristic and trained model** — `_to_features(ContactFeatures)` produces an identical 5-element list in both paths. No divergence between what the heuristic uses and what the model was trained on.
- **Same `/trust-score`, `/predict/churn`, `/predict/ltv` endpoint contract** — The Node API and web components don't change at all in Sprint 23. The improvement is entirely inside the Python microservice.

## Dependencies

- **External:** Historical contact outcome data CSV (generated by exporting Postgres data from 6+ months of beta); `scikit-learn`, `pandas`, `joblib` in `requirements.txt` (added Sprint 18)
- **Internal:** Sprint 18 complete (ML microservice running, heuristic model in place); `services/ml/` Docker container exists

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Not enough labelled churn data after 6 months | Medium | Medium | If <500 labelled examples, heuristic stays active; training is skipped; documented in runbook |
| `joblib` artifact incompatible across scikit-learn versions | Low | Medium | Pin exact `scikit-learn` version in `requirements.txt`; artifact and service always use same version |
| Training script overfits small dataset | Medium | Low | `GradientBoostingClassifier` with `max_depth=3` and 80/20 train-test split; classification report printed so engineer can eyeball before deploying artifact |
